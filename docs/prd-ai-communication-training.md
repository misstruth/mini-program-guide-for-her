# AI 沟通训练计划生成 PRD

## 1. 文档信息

- 产品方向：微信小程序中的 AI 沟通训练功能
- 适用场景：用户输入一句训练诉求，系统自动分析目标、评估当前水平，并生成个性化训练计划
- 示例输入：`提升产品经理沟通能力`
- 目标用户：
  - 普通用户：需要快速得到一版可执行训练方案
  - 会员用户：需要更深度的诊断、动态追踪和训练路径调整

## 2. 背景与机会

当前多数沟通训练产品存在三个问题：

1. 只有泛化建议，没有结合用户真实目标场景
2. 没有先判断当前水平，导致训练计划过难或过浅
3. 训练内容与具体表达句子脱节，用户知道“要这么做”，但不知道“怎么说”

本功能的价值是把一句模糊目标，转成一套可执行、可追踪、可纠偏的训练闭环：

`自然语言目标 -> 能力诊断 -> 个性计划 -> 句子训练 -> 反馈评分 -> 动态调整`

## 3. 产品目标

### 3.1 用户目标

让用户在 1 分钟内拿到一版可信的沟通训练计划，并能马上开始第一轮练习。

### 3.2 业务目标

1. 提升普通用户首次体验完成率
2. 提升会员转化，突出深度诊断和动态 replanning 能力
3. 提升训练连续性，让用户从“看建议”转向“持续练习”

### 3.3 成功指标

- 目标输入后计划生成成功率 > 95%
- 首次计划查看完成率 > 75%
- 首次训练开始率 > 45%
- 7 日训练留存率 > 25%
- 会员方案采纳率 > 普通版 1.5 倍

## 4. 核心用户问题

用户输入“提升产品经理沟通能力”这类目标时，本质上不是要一段文案，而是要 4 个结果：

1. 系统先理解“我到底想提升哪种沟通”
2. 系统判断“我现在是什么水平”
3. 系统告诉“我接下来 7 天 / 14 天 / 28 天应该练什么”
4. 系统给出“在具体场景里我应该怎么说”

## 5. 用户分层与智能模式

### 5.1 普通用户模式

采用：`意图识别 + Agent`

特点：

- 低成本、快响应
- 适合首次体验和轻量训练
- 生成一版标准化计划

典型链路：

1. 识别用户目标意图
2. 识别职业角色、沟通对象、训练主题
3. 从 RAG 中召回对应能力模型、训练模板、场景语料
4. 由 Agent 生成目标、水平判断、训练计划、首批练习句

### 5.2 会员用户模式

采用：`Planner -> Executor -> Replanner`

特点：

- 诊断更细，过程更长
- 可以把复杂目标拆成多个子问题
- 每轮练习后可动态调整计划

典型链路：

1. Planner：拆解训练目标、确定需要补充哪些诊断信息
2. Executor：
   - 执行能力诊断
   - 检索 RAG 知识
   - 生成分阶段训练任务
   - 基于用户作答/练习结果做评分
3. Replanner：
   - 根据训练表现重排优先级
   - 调整难度
   - 增减场景句子包

## 6. 目标场景定义

以“提升产品经理沟通能力”为例，系统不能只输出一个笼统主题，而应落到具体沟通场景：

- 需求澄清
- 跨部门协同
- 向上汇报
- 推进延期项目
- 冲突反馈
- 用户访谈追问
- 与研发对齐优先级
- 会议表达与总结

系统应将用户的模糊目标自动映射到 1~3 个核心训练场景。

## 7. 功能范围

### 7.1 本期必须实现

1. 支持用户输入自然语言训练目标
2. 自动识别训练主题、职业角色、场景方向
3. 自动生成训练目标
4. 自动分析当前水平
5. 自动生成 7 天 / 14 天训练计划
6. 为每个训练任务关联具体句子练习
7. 支持普通版和会员版两套链路

### 7.2 后续可扩展

1. 支持语音练习
2. 支持角色扮演对话
3. 支持训练过程中的实时插话纠偏
4. 支持企业岗位专属知识库

## 8. 用户流程

### 8.1 首次生成计划

1. 用户输入一句目标，例如：`提升产品经理沟通能力`
2. 系统追问或自动补全上下文：
   - 当前岗位
   - 常见沟通对象
   - 最大沟通难点
   - 期望周期
3. 系统生成：
   - 目标定义
   - 当前水平判断
   - 差距分析
   - 训练计划
   - 第一批场景句子
4. 用户开始第一节训练

### 8.2 训练中动态调整

1. 用户完成某轮句子表达或对话练习
2. 系统按能力维度评分
3. 会员模式触发 replanner
4. 生成下一轮更合适的任务与句子包

## 9. 核心方案设计

## 9.1 输入理解层

用户原始输入通常很短，因此系统需要做“目标理解扩写”。

建议识别以下字段：

- `goal_text`：原始目标文本
- `role`：角色，如产品经理、运营、销售
- `domain`：训练领域，如职场沟通、销售沟通、管理沟通
- `scenario_tags`：场景标签
- `capability_tags`：能力标签
- `target_object`：沟通对象，如老板、研发、客户
- `urgency`：短期解决问题 / 长期提升
- `membership_type`：普通 / 会员

示例：

```json
{
  "goal_text": "提升产品经理沟通能力",
  "role": "产品经理",
  "domain": "职场沟通",
  "scenario_tags": ["需求澄清", "跨部门协同", "向上汇报"],
  "capability_tags": ["结构化表达", "追问澄清", "冲突降温", "结论先行"],
  "target_object": ["研发", "设计", "上级"],
  "urgency": "长期提升"
}
```

## 9.2 当前水平分析层

“当前水平”不能由模型一句话拍脑袋给出，建议采用 4 类输入联合判断：

1. 用户自评
2. 启动问答
3. 历史训练数据
4. 真实表达样本

### 诊断维度建议

- 目标清晰度：表达时是否知道自己要什么
- 结构化程度：是否会先结论后细节
- 澄清追问能力：是否能问到关键约束
- 共情与语气：是否能降低对抗感
- 推进能力：是否能把讨论推进到下一步
- 复盘能力：是否能总结问题与改进点

### 水平分层建议

- L1：意识型，只知道自己沟通弱，但不会拆解问题
- L2：模仿型，会套句式，但不稳定
- L3：场景型，常见场景基本能应对
- L4：策略型，能根据对象与局势调整表达
- L5：引导型，能主动主导复杂沟通

### 当前水平输出方式

不要只输出一句“你当前水平中等”，而要输出结构化诊断：

```json
{
  "current_level": "L2",
  "confidence": 0.74,
  "dimension_scores": {
    "目标清晰度": 72,
    "结构化表达": 58,
    "澄清追问": 49,
    "共情语气": 61,
    "推进能力": 55
  },
  "strengths": ["有基本表达意愿", "能概述问题背景"],
  "weaknesses": ["追问不够深入", "结论不够前置", "句子容易绕"],
  "evidence": [
    "在需求沟通样本中缺少对时间、优先级、验收标准的追问",
    "在汇报表达中先讲过程，后讲结论"
  ]
}
```

## 9.3 训练计划生成层

训练计划建议分 3 层输出，而不是只给一个大段文本。

### 第一层：计划摘要

给用户一个 10 秒内能看懂的答案：

- 你的目标是什么
- 你当前在哪个水平
- 你优先要补哪 2~3 项
- 这期训练会持续多久

### 第二层：周期计划

按周或按阶段输出训练路径。

例如 14 天计划：

- 第 1 阶段：建立结构化表达
- 第 2 阶段：训练关键追问
- 第 3 阶段：训练推进与确认
- 第 4 阶段：真实场景模拟

### 第三层：单次训练任务

每次训练都应该明确：

- 练什么能力
- 在什么场景下练
- 用哪些句子练
- 如何判断练得好不好

## 9.4 训练计划建议输出结构

推荐前后端统一使用结构化 JSON，前端再渲染成卡片。

```json
{
  "plan_id": "plan_001",
  "goal_summary": {
    "title": "提升产品经理核心沟通能力",
    "description": "重点提升需求澄清、跨部门协同、向上汇报三类沟通能力"
  },
  "current_assessment": {
    "level": "L2",
    "main_gap": ["结构化表达", "澄清追问", "推进确认"]
  },
  "milestones": [
    "第 3 天能用结论先行表达需求问题",
    "第 7 天能围绕需求提出 3 个关键追问",
    "第 14 天能完成一次完整的项目进度汇报"
  ],
  "cycles": [
    {
      "cycle_no": 1,
      "name": "结构化表达建立",
      "days": [1, 2, 3],
      "focus_skills": ["结论先行", "信息分层"],
      "tasks": [
        {
          "task_id": "task_101",
          "scenario": "向研发说明需求变更",
          "objective": "先说结论，再说原因和影响",
          "sentence_pack_ids": ["sp_001", "sp_002"],
          "evaluation_rules": ["是否先给结论", "是否说明影响范围", "是否提出下一步"]
        }
      ]
    }

```

## 10. 训练计划怎么输出

这里建议明确采用“结构化数据 + 可读卡片”的双层输出。

### 10.1 面向模型和系统的输出

用 JSON 存储，原因：

1. 便于前端渲染
2. 便于后端存档和版本管理
3. 便于会员模式后续 replanning
4. 便于统计计划采纳率和完成率

### 10.2 面向用户的输出

前端建议展示为 5 块卡片：

1. 目标卡
   - 你要练什么
   - 为什么练这个
2. 水平卡
   - 当前等级
   - 优势
   - 短板
3. 路线图卡
   - 7 天或 14 天阶段目标
4. 每日任务卡
   - 今天练习的场景
   - 练习目标
   - 开始练习按钮
5. 句子练习卡
   - 推荐表达
   - 错误表达
   - 改写建议

### 10.3 输出粒度建议

不要一次给用户 28 天全部细节，建议：

- 首屏展示全局目标 + 当前水平 + 最近 3 天任务
- 进入计划详情后再看完整周期
- 每次训练开始时展示当前任务对应的句子包

这样可以减少信息压迫感。

## 11. 句子怎么关联

这是这个功能最关键的部分。建议不要直接把“训练计划”和“句子库”做成松散关系，而是要建立一套“能力图谱驱动的句子关联机制”。

## 11.1 关联原则

一句话不是孤立训练单元，它至少同时属于以下 5 个维度：

1. 所属场景
2. 所属意图
3. 所属能力点
4. 难度等级
5. 常见错误类型

例如这句：

`我先确认一下，这次需求的目标用户是谁，核心成功指标是什么？`

它可以关联为：

- 场景：需求澄清
- 意图：追问关键约束
- 能力点：澄清追问
- 难度：L2
- 适用角色：产品经理
- 常见错误：问题太泛、没有问到验收标准

## 11.2 建议的知识库建模

### 实体 1：训练目标

- `goal_id`
- `goal_name`
- `domain`
- `role_tags`

### 实体 2：场景

- `scenario_id`
- `scenario_name`
- `description`
- `target_objects`

### 实体 3：能力点

- `skill_id`
- `skill_name`
- `skill_dimension`
- `level_range`

### 实体 4：句子

- `sentence_id`
- `text`
- `scene_tags`
- `intent_tags`
- `skill_tags`
- `difficulty`
- `tone`
- `good_example`
- `bad_example`
- `rewrite_tips`

### 实体 5：训练任务

- `task_id`
- `plan_id`
- `scenario_id`
- `skill_ids`
- `sentence_pack_ids`

## 11.3 推荐关联链路

建议按下面的链路做，而不是直接“目标 -> 句子”：

`目标 -> 场景 -> 能力点 -> 任务 -> 句子包`

原因：

1. 一个目标会覆盖多个场景
2. 一个场景会对应多个能力点
3. 一个能力点在不同等级下对应不同句子
4. 后续做 replanner 时，调整的是任务和能力点，不是直接乱换句子

### 示例

用户目标：`提升产品经理沟通能力`

系统拆解：

- 场景 A：需求澄清
- 场景 B：跨部门推进
- 场景 C：向上汇报

场景 A 对应能力点：

- 关键追问
- 需求复述确认
- 风险澄清

然后每个能力点下挂句子包：

- 入门句型
- 进阶句型
- 反面句型
- 改写练习

## 11.4 句子包怎么设计

建议不要只存单句，建议存“句子包”。

每个句子包包含：

- 标准表达 3~5 句
- 常见错误表达 2~3 句
- 改写练习 2~3 题
- 场景上下文
- 评价标准

示例：

```json
{
  "sentence_pack_id": "sp_001",
  "scenario": "需求澄清",
  "skill": "关键追问",
  "difficulty": "L2",
  "context": "研发说需求描述不清，无法排期",
  "good_sentences": [
    "我先确认一下，这个需求最核心想解决的用户问题是什么？",
    "如果本期只能保一个目标，优先级最高的是哪一项？",
    "验收时我们用什么指标判断这个需求是成功的？"
  ],
  "bad_sentences": [
    "那你们先看看能不能做。",
    "需求大概就是这个意思。"
  ],
  "rewrite_tasks": [
    {
      "source": "这个功能你们先做一下吧。",
      "target_skill": "澄清追问"
    }
  ],
  "rubric": ["是否问到目标", "是否问到优先级", "是否问到验收标准"]
}
```

## 11.5 句子关联的实际检索逻辑

建议 RAG 检索条件至少包含：

- `role`
- `scenario`
- `skill`
- `difficulty`
- `current_level`
- `recent_errors`

检索逻辑示例：

1. 先根据目标确定核心场景
2. 再根据当前短板确定优先能力点
3. 再根据能力点和等级召回句子包
4. 若用户最近在“追问不够具体”上持续低分，则优先召回该错误类型对应的句子包

这会比单纯语义召回稳定很多。

## 12. RAG 设计建议

不要只把文章丢进向量库。这个场景更适合“结构化知识 + 向量检索”的混合 RAG。

### 建议纳入 RAG 的内容

1. 沟通能力模型
2. 场景定义库
3. 句子包库
4. 高质量对话示例
5. 评分 rubric
6. 常见错误表达与修正策略
7. 行业/岗位专属表达模板

### 检索优先级建议

1. 结构化过滤
2. 向量召回
3. rerank
4. Agent 生成

也就是先把不相关场景过滤掉，再做语义相似召回，否则容易把销售沟通、管理沟通、客服沟通混在一起。

## 13. Agent 设计建议

## 13.1 普通用户 Agent

建议单 Agent 负责以下产出：

- 识别目标
- 生成当前水平初判
- 生成训练计划摘要
- 生成第一批训练句子

输出要克制，优先保证“可执行”。

## 13.2 会员 Planner

Planner 主要负责：

- 拆解目标
- 规划诊断步骤
- 确定训练阶段
- 选择需要调用的 executor

### 建议 planner 输出

```json
{
  "sub_goals": ["提升需求澄清能力", "提升跨部门推进表达", "提升向上汇报结构化能力"],
  "diagnosis_needs": ["评估结论先行能力", "评估追问深度", "评估冲突表达"],
  "execution_plan": [
    "召回产品经理沟通知识库",
    "生成能力诊断题",
    "生成 14 天计划",
    "绑定句子包与评分 rubric"
  ]
}
```

## 13.3 Executor

Executor 可以按职责拆分：

- Diagnosis Executor：做水平评估
- Plan Executor：生成训练计划
- Sentence Executor：召回/生成句子包
- Review Executor：对用户训练结果评分

## 13.4 Replanner

触发条件建议：

- 用户连续 2 次某维度评分低于阈值
- 某训练任务连续未完成
- 用户主动反馈“太难/太简单/不贴合”

Replanner 输出：

- 降难 / 升难
- 更换场景
- 加强某能力点
- 插入补救任务

## 14. 前端交互建议

首页或生成页建议流程：

1. 输入框：`你想提升哪方面沟通能力？`
2. 快捷标签：
   - 向上汇报
   - 跨部门沟通
   - 需求澄清
   - 冲突沟通
3. 生成中：
   - 正在识别目标
   - 正在分析当前水平
   - 正在生成训练路线
4. 结果页：
   - 你的目标
   - 当前水平
   - 本期路线图
   - 今日开始任务

## 16. 数据存储建议

至少应存以下对象：

- 训练计划表 `training_plans`
- 训练任务表 `training_tasks`
- 句子表 `sentences`
- 句子包表 `sentence_packs`
- 能力点表 `skills`
- 场景表 `scenarios`
- 用户训练记录表 `training_records`
- 用户能力评分快照表 `user_skill_snapshots`

## 17. 风险与注意事项

1. 如果只靠 LLM 直接生成，计划稳定性会差
2. 如果没有句子包和 rubric，训练会沦为泛文案
3. 如果不先做水平评估，用户会觉得内容“不像给我的”
4. 如果会员链路过长，生成耗时可能影响转化，需要异步和流式展示

## 18. MVP 建议

第一版建议先做稳，不要一开始就追求特别复杂的 agent 编排。

### MVP 范围

1. 普通用户：
   - 意图识别
   - RAG 召回
   - 单 Agent 生成计划
2. 会员用户：
   - 增加诊断问答
   - 增加 planner / replanner
   - 每次训练后更新下一轮任务
3. 句子库：
   - 先覆盖产品经理 4 个高频场景
   - 每个场景先做 10~20 个句子包

## 19. 最终建议

这个功能的核心不是“让 AI 写一段训练建议”，而是把训练真正产品化：

1. 目标要结构化
2. 水平判断要有证据
3. 计划要分层输出
4. 句子要通过场景和能力点强关联
5. 会员价值要体现在动态 replanning，而不是只是“更长的答案”

如果按这个方向做，你的 Agent + RAG 才会有明显产品差异，而不是普通问答包装。
